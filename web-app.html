Carty Web App — Functional Requirements

Test specification for the Carty web app: the public payment link page, login, and the account dashboard. Tick each acceptance criterion as you verify it — progress is saved in this browser.

Version 1.0 Date 11 Jul 2026 Audience QA App carty_new_website API api.razz.ng

1 · Overview & test setup

The web app extends the marketing site with a transactional surface: a public payment link page (/pay/[code]) with item details, a QR code, and pay-from-wallet checkout; a login flow (/login) for existing Carty accounts; and an authenticated dashboard (/dashboard) with wallet, payment links, orders, transactions, and profile.

Money is escrow-first: the buyer's wallet is debited at checkout and the seller is credited only after the buyer confirms delivery. All amounts are stored in kobo by the API and displayed in naira (API 150000 → ₦1,500).

Actors

ActorDescription
VisitorNot logged in. Can view payment link pages and marketing pages only.
BuyerLogged-in user opening someone else's payment link or viewing orders they placed.
SellerLogged-in user who created payment links / receives orders. Any account can be both buyer and seller — never both on the same order.
Pre-KYC userLogged-in user without identity (Dojah) verification — has no virtual account number.

2 · Functional requirements

FR-AUTH — Authentication & session

FR-AUTH-1

Email/password login

/login accepts email + password and authenticates against the Carty API.

  1. Valid credentials (no 2FA) log the user in and redirect to /dashboard (or the ?next= target).
  2. Invalid credentials show the API error (e.g. "email or password is incorrect") without navigation.
  3. Both fields are required; the browser blocks empty submission.
FR-AUTH-2

Two-factor (OTP) step

  1. After a correct password on a 2FA account, the form switches to a one-time-code input with a message saying where the code was sent (the email address you provided during registration).
  2. Correct OTP completes login; wrong OTP shows an error and allows retry.
  3. "Use a different account" returns to the email/password step.
FR-AUTH-4

Route protection & redirects

  1. Visiting any /dashboard* URL logged-out redirects (307) to /login?next=<that URL>.
  2. After login, the user lands on the original next target.
  3. next values that are not same-site relative paths (e.g. https://evil.com, //evil.com) are ignored — fallback is /dashboard.
  4. Visiting /login while already logged in redirects straight to the target.
FR-AUTH-5

Logout

  1. "Log out" in the dashboard nav clears the session cookies and returns to /login.
  2. After logout, /dashboard redirects to login and the header reverts to "Log in".
FR-AUTH-6

Header session state

  1. Logged out: the site header shows a "Log in" link.
  2. Logged in: the header shows an initials avatar + first name linking to /dashboard — on all pages using the shared header, including the landing page.
FR-AUTH-7

No web registration

  1. The login page links prospective users to /download ("Get the Carty app") — there is no sign-up form on the web.

FR-PAY — Public payment link page (/pay/[code])

FR-PAY-1

Link details display

  1. A valid, active code renders: title, price in ₦, "Active payment link" badge, description (if any), delivery method, pickup area (if any), and product photos.
  2. Photos render as a gallery: main image plus a thumbnail strip when more than one; clicking a thumbnail swaps the main image.
  3. An unknown/invalid code returns the site's 404 page.
  4. Page metadata (tab title / link preview) contains the item title and price; the first photo is the preview image.
FR-PAY-2

QR code

  1. The QR encodes SITE_URL/pay/<code> — scanning with a phone camera opens the same page.
  2. Scanning with the Carty mobile app's link scanner resolves the same payment link (the app extracts the trailing code).
  3. The QR sits on a white tile and scans in both dark and light site themes.
  4. The copy button copies the page URL and shows "Copied!" feedback.
FR-PAY-3

Visitor (logged-out) state

  1. The payment panel explains escrow and shows "Log in to pay" → /login?next=/pay/<code> (returns here after login) and "Get the app".
  2. No checkout fields are visible while logged out.
FR-PAY-4

Inactive link

  1. A link with status inactive shows "Link no longer active"; no checkout is possible for any user.
FR-PAY-5

Seller viewing own link

  1. The link's owner sees "This is your payment link" (share guidance + "View my orders") instead of checkout — a seller cannot buy from their own link (also enforced by the API).

FR-CHK — Checkout (logged-in buyer)

FR-CHK-1

Checkout form

  1. Shows the buyer's wallet balance chip.
  2. Fields: quantity (min 1), receiver's name and phone (pre-filled from profile), delivery address, state and city/LGA selects, optional delivery note.
  3. City select is disabled until a state is chosen; changing state resets city.
FR-CHK-2

Delivery fee

  1. Selecting a city fetches and shows the delivery fee (₦1,500 default).
  2. Until a city is chosen, the fee row reads "Select city"; the Pay button is disabled during fee lookups.
  3. Changing city quickly in succession never leaves a fee from a previously selected city on screen.
FR-CHK-3

Extra inspection (service fee)

  1. If the API returns active extra-cost options, "Request extra inspection" reveals them with prices.
  2. Selected options add an "Extra inspection" line and increase the total.
  3. Unchecking the master checkbox clears selections and removes the charge.
FR-CHK-4

Order summary & totals

  1. Summary shows item × quantity, delivery fee, inspection (if any), and total = item×qty + fee + inspection.
  2. Changing quantity updates the item line and total immediately.
FR-CHK-5

Insufficient balance

  1. If total > wallet balance: a warning shows the balance and points to funding in the app; the Pay button is disabled.
FR-CHK-6

Placing the order

  1. "Pay ₦X from wallet" debits the wallet, creates the order in escrow, and shows: "Payment in escrow", amount paid, order number, "Track order" and "Go to dashboard".
  2. Missing address or state/city shows inline errors without submitting.
  3. API failures (e.g. "insufficient balance", "payment link is not active") surface as readable errors; the form stays usable.
  4. After success: balance is reduced by the total; the order appears in Orders as "In escrow"; the debit appears in Transactions.

FR-DASH — Dashboard overview (/dashboard)

FR-DASH-1

Wallet card (mirrors the mobile app)

  1. Has account: balance, account number with copy button ("Copied!" feedback), account name and bank, funding hint.
  2. KYC-verified, no account yet: "Your virtual account is not ready yet." with a Get account number button; success refreshes the card with the new details.
  3. Not KYC-verified: "Complete your identity verification in the Carty app to get an account number for funding your wallet." with Verify in the app → /download. No verification flow runs on the web.
  4. The eye toggle hides/shows the balance (••••••••).
FR-DASH-2

Quick stats & recent orders

  1. "Active orders" counts orders not completed/cancelled; "Payment links" counts the user's links; both tiles navigate to their sections.
  2. Up to 5 most recent orders show title, order number, date, amount, and status badge; each row opens the order detail.
  3. With no orders, an empty state invites the user to share a payment link.

FR-LINK — Payment links (/dashboard/links)

FR-LINK-1

List

  1. Each link shows thumbnail (or initial placeholder), title, price, created date, status badge.
  2. "Copy" copies SITE_URL/pay/<code>; "View" opens the public page.
  3. Empty state offers "Create your first link".
FR-LINK-2

Create link (/dashboard/links/new)

  1. Required: title and price (naira, decimals allowed, > 0). Optional: description, delivery method (rider/pickup), pickup area, photos.
  2. Invalid price (0, negative, non-numeric) is rejected with an error before submission.
  3. On success the user is redirected to the new link's public page /pay/<code> (doubles as the share screen with QR).
  4. The new link then appears in the links list and in the mobile app.
FR-LINK-3

Product images

  1. "Add" opens the file picker offering only .jpg .jpeg .png .gif .bmp (the API's allowlist).
  2. Selected images show preview thumbnails immediately, each with ✕ to remove.
  3. Maximum 5 images; the Add tile disappears at 5 and returns after removal. Over-selection adds only up to the cap.
  4. Picking the same file twice in a row works (the input resets after each pick).
  5. An unsupported extension is rejected client-side with a message naming the file.
  6. Uploaded photos appear on the public page and in the links list after creation.

FR-ORD — Orders (/dashboard/orders)

FR-ORD-1

Orders list

  1. All orders where the user is buyer or seller, newest first: title, role tag ("Buying"/"Selling"), order number, date, total, status badge.
  2. Status labels: in_escrow → "In escrow", assigned → "Rider assigned", picked_up → "Picked up", delivered → "Delivered", completed → "Completed", disputed → "Disputed", cancelled → "Cancelled".
FR-ORD-2

Order detail

  1. Shows item (with photo), quantity, placed date, and amount breakdown (item, delivery fee, service/inspection, total).
  2. Shows delivery details: receiver, phone, address (with LGA/state), note.
  3. Shows the progress timeline newest-first (e.g. "Payment held in escrow" after checkout) with dates.
  4. Delivery code visibility: the buyer sees their delivery code with guidance to share only on receipt. The seller must never see the delivery code.
  5. Accessing an order the user is not a party to returns 404.
FR-ORD-3

Confirm delivery (buyer only)

  1. Available to the buyer while the order is open (not completed/cancelled/disputed).
  2. The correct delivery code marks the order Delivered → Completed and shows "payment has been released to the seller".
  3. A wrong code shows "Invalid delivery code" and changes nothing.
  4. After confirmation: the seller's balance increases by the item amount, the status badge reads "Completed", and the timeline gains "Order Completed".
  5. The seller has no confirm-delivery control; the API also rejects a seller completing an order ("Only the buyer can complete this order.").
FR-ORD-4

Disputes

  1. Buyer or seller can open a dispute on an open order; reason and description are required.
  2. On success the order refreshes with a confirmation notice; status becomes "Disputed".
FR-ORD-5

Rating

  1. Available only once the order is Completed.
  2. Requires a 1–5 star selection (comment optional); zero stars shows an error.
  3. Success shows a thank-you notice.

FR-TXN — Transactions (/dashboard/transactions)

FR-TXN-1

Transactions list

  1. Up to the 50 most recent non-pending transactions, newest first: description, transaction ID, date, status badge, signed amount (− for debits, + green for credits).
  2. A Carty order payment appears as a "successful" debit immediately after checkout.
  3. Empty state renders when the account has no transactions.

FR-PROF — Profile (/dashboard/profile)

FR-PROF-1

Profile view & edit

  1. Shows photo (or initials), full name, verified badge when KYC is complete, @tag, and read-only email and phone (visually disabled).
  2. First/last name are editable; "Save changes" is disabled until something changes and re-disables after a successful save ("Profile updated.").
  3. Photo can be changed (.jpg .jpeg .png .gif .bmp) with an immediate preview; the saved photo persists after reload and shows in the mobile app.
  4. A banner promotes the app for everything else ("Do more with the Carty app…") with a Get-the-app button.
  5. Name changes appear in the dashboard greeting and header after refresh.

3 · Cross-cutting requirements

NFR-1

API mediation

  1. The browser never calls the Go API directly — all traffic goes through the web app's origin (/api/auth/*, /api/proxy/*).
  2. The proxy exposes only carty/, me/, transaction, notification paths; anything else (e.g. /api/proxy/auth/login) returns 404.
NFR-2

Error resilience

  1. API unreachable: /pay/[code] returns 404; dashboard pages render with empty data rather than crashing.
  2. Form submissions during an outage show "Could not reach Carty. Please try again."
NFR-3

Theme & responsiveness

  1. All new pages render correctly in dark (default) and light themes.
  2. On mobile viewports the checkout panel stacks below the item details and the dashboard nav becomes a horizontal scroll row.
NFR-4

Currency display

  1. Every amount = API kobo ÷ 100, formatted with ₦ and thousands separators; whole-naira amounts show no decimals.
NFR-5

SEO / crawling

  1. robots.txt disallows /dashboard and /api/; payment pages remain crawlable.

4 · Out of scope — do not raise as defects

  • Registration, password reset, and KYC/Dojah verification on the web (app-only; web links out to /download).
  • Wallet funding by card, withdrawals, transfers (funding is by bank transfer to the virtual account, or in the app).
  • Changing email/phone on the web.
  • Rider-facing and admin features.
  • Payment link editing/deactivation from the web (create + share only).
  • Push/real-time notifications and live rider map tracking.

5 · Known behaviours useful for QA

  • Delivery fee defaults to ₦1,500 for any city without a matching extra-cost record.
  • The checkout service fee is the sum of selected "extra inspection" options (₦0 when none).
  • Orders skip pending_payment: wallet checkout puts them directly into in_escrow.
  • The API returns shareable links as https://carty.ng/pay/<code> (base overridable with CARTY_WEB_URL), so links created in the mobile app open the web page under test.
  • Session cookies last ~30 days, but logout (web or app) or a password change invalidates the token server-side — the next dashboard request redirects to login.
